Lucode is organized around **tasks** as the only top-level entity. Each task owns a base git worktree branching from your repository, carries planning artifacts (Spec, Plan, Summary), and progresses through a series of stages. Stage runs spawn slot sessions under the task where agents do focused work.

## Tasks

<Card title="What is a Task?" icon="laptop-code">
  A task is the unit of work in Lucode. It owns a dedicated git worktree, a branch, and a set of artifacts (the request body, plus any Spec/Plan/Summary documents the workflow produces). Stage runs under the task spawn slot sessions where agents work side-by-side with you in dual terminals.
</Card>

<CardGroup cols={2}>
  <Card title="Isolated Worktree" icon="git-alt">
    Each task gets its own git worktree and branch
  </Card>
  <Card title="Dual Terminals" icon="terminal">
    Top terminal for the agent, bottom terminal for manual commands
  </Card>
  <Card title="Instant Switching" icon="arrows-rotate">
    Switch between tasks instantly—terminals stay alive in the background
  </Card>
  <Card title="Sidebar Organization" icon="list">
    Tasks appear in the sidebar grouped by stage section
  </Card>
</CardGroup>

<Tip>
  Every task worktree is its own copy of the repository on disk. Keep tasks short-lived and clean up finished work so the `.lucode/worktrees/` folder doesn't grow without bound.
</Tip>

### Renaming Tasks

Double-click on a task name in the sidebar to rename it. The git branch will be updated to match the new name.

## Task Artifacts

A task carries its planning context as **artifacts** stored alongside the worktree:

<CardGroup cols={2}>
  <Card title="Spec" icon="file-lines">
    The task’s request body—what you want done. This is the agent’s initial prompt when a stage run launches.
  </Card>
  <Card title="Plan" icon="list-check">
    A Plan artifact captures the implementation approach generated during planning stages.
  </Card>
  <Card title="Summary" icon="bookmark">
    A Summary artifact records what was done after the work finishes.
  </Card>
  <Card title="Recycle a task" icon="rotate">
    If a run isn’t working out, cancel it; the Spec artifact stays so you can relaunch later with new instructions.
  </Card>
</CardGroup>

<Info>
  Specs, Plans, and Summaries are artifacts of a task—not standalone entities. There is no separate “Specs” collection in v2.
</Info>

## Orchestrator

<Card title="The Control Panel" icon="tower-control" color="#22d3ee">
  The orchestrator is a singleton non-task surface that runs directly on your primary branch—typically `main`, but it can be a long-lived feature branch too—giving you a stable workspace for planning and coordination.
</Card>

### Orchestrator Features

<AccordionGroup>
  <Accordion title="Coordination & Planning">
    Use the orchestrator to triage incoming work, draft new tasks, and supervise multiple agents at once.
  </Accordion>

  <Accordion title="MCP REST API">
    The orchestrator powers automation through the MCP REST API, enabling external tools to create and manage tasks.
  </Accordion>

  <Accordion title="Dual Terminals">
    Like a slot session, the orchestrator has an agent terminal on top and your shell below.
  </Accordion>
</AccordionGroup>

<Warning>
  Because the orchestrator operates inside your primary branch (no separate worktree), any commands you run there change that branch immediately. Use tasks when you need isolation.
</Warning>

### Accessing the Orchestrator

<CodeGroup>
```bash Keyboard Shortcut
⌘1
```

```plaintext Sidebar
Click "Repository (Orchestrator)" at the top of the sidebar
```
</CodeGroup>

## Task Stages

Tasks progress through the following stages:

<Steps>
  <Step title="Draft" icon="file-pen">
    The task is being captured. The Spec artifact is being written; the agent hasn’t been asked to do anything yet.
  </Step>

  <Step title="Ready" icon="circle-check">
    The Spec is ready for the agent to start working from.
  </Step>

  <Step title="Brainstormed" icon="lightbulb">
    Initial exploration / brainstorming has happened. Planning can begin.
  </Step>

  <Step title="Planned" icon="list-check">
    A Plan artifact exists. Implementation can begin.
  </Step>

  <Step title="Implemented" icon="code">
    The implementation work is complete in the task’s worktree.
  </Step>

  <Step title="Pushed" icon="upload">
    Changes have been pushed to the remote (typically as a PR or branch).
  </Step>

  <Step title="Done" icon="check">
    The task is finished—merged or otherwise resolved.
  </Step>
</Steps>

<Info>
  **Cancellation is orthogonal.** A task can be cancelled at any stage; cancellation is tracked separately (`cancelled_at`) rather than being a stage of its own.
</Info>

### Stage Transitions

<Info>
  Stage transitions are explicit and user-controlled. The app never automatically advances a task’s stage.
</Info>
