When orchestrating multiple agents you get the best results by laying a shared contract first, then letting specialists branch off in parallel. Think of it like traditional engineering: define the API surface and domain seams before frontend, backend, and data teams sprint ahead. The playbooks below help you sequence that work so tasks stay short-lived, merge conflicts stay rare, and every agent knows the rules of the codebase.

## Scaffold ‚Üí Swarm ‚Üí Stabilize

![Diagram showing layered scaffolding for an email-report application: top layer (frontend/controller), middle layer (VehicleQueryService) coordinating downstream services, and bottom layer composed of repositories, parsers, and generators feeding the service.](/images/scaffold.png)

The dashed bands illustrate how the scaffolding phase isolates concerns: first establish the orchestration layer (green) that defines the API contract, then set up infrastructure primitives such as repositories and parsers (blue). Once this walking skeleton is in place, feature-specific surfaces like the frontend (red) can be delegated to independent agents without tripping over one another. For a concise primer, see Matt Blodgett‚Äôs summary of Cockburn‚Äôs walking skeleton pattern (September 28, 2020) at [mattblodgett.com](https://www.mattblodgett.com/2020/09/start-with-walking-skeleton.html).

<Steps>
  <Step title="Scaffold in the orchestrator">
    Use a task (often coordinated from the orchestrator) to define project inputs/outputs and CLI affordances. Document how to run tests, builds, lint, and seed data; verify each command headlessly in the orchestrator terminals; and capture everything in an `AGENTS.md` so every future agent starts with the same contract.
  </Step>

  <Step title="Bootstrap the walking skeleton">
    Launch an initial task to commit the scaffolding: repository layout, shared config, domain models, controller/service shells, and automation hooks. This follows the walking skeleton approach: ship a minimal end-to-end slice that proves the pipeline works so every downstream task depends on clear interfaces instead of volatile files.
  </Step>

  <Step title="Swarm on features">
    Break work into focused tasks (for example, "Billing webhook handler", "Settings sidebar UI", "Analytics ingestion job") and run agents in parallel. Because the skeleton already expresses the shared contracts, each task can focus on its slice without churn in upstream files.
  </Step>

  <Step title="Stabilize and integrate">
    Use the second terminal pane for tests and linting, review diffs, and merge with `‚åò‚áßM` or open a PR once everything is green. Short-lived task worktrees keep disk usage in check and allow rapid iteration.
  </Step>

  <Step title="Recycle unfinished ideas">
    If a concept needs more thought, cancel the task; its archived Spec artifact stays so you can launch a fresh task with new context later.
  </Step>
</Steps>

![Diagram showing the swarm phase: orchestrator/main branches into separate agent tasks (Email Generation, PDF Generation, Frontend) that each work in isolated worktrees.](/images/swarm.png)

As soon as the walking skeleton lands, branch from that commit (whether it lives on `main` or a feature branch) and launch agents for each missing capability. Every task owns its worktree, so it can install dependencies, run tests, and iterate on its feature without stepping on the others. Once a slice is green and reviewed, merge it back, recycle any unfinished tasks, and keep the swarm moving.

## Integrate, Review, Repeat

After each task delivers a passing feature‚Äîunit tests green, manual QA satisfied, review notes resolved‚Äîchoose the merge strategy that fits your flow:

- **Pull request (`‚åò‚áßP`)** for collaborative review, or
- **Direct merge (`‚åò‚áßM`)** when you want to squash the worktree back into the skeleton branch yourself.

Resolving conflicts centrally keeps history clean: pull the branch into the orchestrator (or the original skeleton branch), fix any mismatches there, then merge. Work through tasks until they‚Äôre cleared; each completion frees disk space because the task‚Äôs worktree can be cancelled.

Need hands-on edits? Press **Open** (`‚åò‚áßO`) on the task card, jump to your preferred IDE, and patch the worktree manually‚Äîagents and humans can collaborate on the same isolated branch. If a feature proves too complex, cancel the task and create a fresh one with a clearer Spec artifact later.

Once the integrations land, inspect the product and backlog. Any new bugs, refactors, or missing tests go straight into fresh tasks so you can swarm again‚Äîtreating the agents like a full squad of engineers you direct through tasks, reviews, and tests to hit the exact quality bar you expect.

## Continuous Maintenance Loop

![Diagram showing merge and repeat workflow: multiple feature branches (email, PDF, frontend) converge into the orchestrator/main branch, which then fans out into new maintenance branches (fix UTF-8, add unit tests, refactor frontend).](/images/merge_and_repeat.png)

After integrating the first wave of feature tasks into main, you can immediately create new maintenance tasks for bug fixes, regression tests, or refactors‚Äîkeeping the cycle moving without bloating a single worktree.

<Steps>
  <Step title="Collect fixes as tasks">
    Log production bugs, refactors, and TODOs as lightweight tasks so they remain ready-to-run cards in the backlog.
  </Step>

  <Step title="Launch focused agents">
    Start one task per fix (e.g., "Resolve UTF-8 parsing bug", "Tighten PDF export pipeline", "Rename legacy API module").
  </Step>

  <Step title="Validate locally">
    Reproduce issues, run targeted tests in the second terminal, and annotate findings directly in the task‚Äôs Spec artifact.
  </Step>

  <Step title="Review, merge, recycle">
    Approve the diff, then merge or open a PR. Cancel anything unresolved and create a fresh task with new context to revisit later.
  </Step>
</Steps>

## Multi-Agent Tasks

Run multiple agents on the same problem to compare approaches or pick the best result.

<Tabs>
  <Tab title="Same Agent, Multiple Versions">
    Run 2‚Äì4 instances of the same agent in parallel. Each version works independently on the same prompt.

    ![Version selector dropdown showing 1-4 versions and Use Multiple Agents option](/images/versions.png)

    <Steps>
      <Step title="Select version count">
        In the New Task dialog, click the version dropdown and choose **2 versions**, **3 versions**, or **4 versions**
      </Step>

      <Step title="All versions start together">
        Lucode creates linked tasks (e.g., `my_task`, `my_task_v2`) each running the same agent in its own worktree
      </Step>

      <Step title="Compare and choose">
        Review each version's output, pick the best result, and merge. Cancel the others with `‚åòD`.
      </Step>
    </Steps>
  </Tab>

  <Tab title="Different Agents">
    Run different agents (e.g., Claude + Codex) on the same problem to compare their approaches.

    ![Multi-agent setup showing agent selection with checkboxes and instance count selectors](/images/multiple-agent-versions.png)

    <Steps>
      <Step title="Open multi-agent mode">
        In the New Task dialog, click the version dropdown and select **Use Multiple Agents**
      </Step>

      <Step title="Configure agents">
        Check the agents you want and set instance counts (up to 4 total). For example: 1x Claude, 1x Codex, 2x OpenCode.
      </Step>

      <Step title="All agents start together">
        Lucode creates linked tasks each running its configured agent in its own worktree with the same prompt
      </Step>

      <Step title="Compare and choose">
        Review each agent's output, pick the best result, and merge. Cancel the others with `‚åòD`.
      </Step>
    </Steps>
  </Tab>
</Tabs>

## Parallel Development Tips

<Steps>
  <Step title="Create multiple tasks">
    Spin up separate tasks for each initiative (e.g., `‚åòN` ‚Üí "Add authentication", `‚åòN` ‚Üí "Fix checkout bug", `‚åòN` ‚Üí "Refactor database layer").
  </Step>

  <Step title="Switch deliberately">
    Use `‚åò1-9` to jump between tasks or <code>‚å•‚áß`</code> / <code>‚å•`</code> (`‚åò‚Üë/‚Üì` also works) to cycle through sidebar items.
  </Step>

  <Step title="Monitor all at once">
    Each agent runs inside its own task worktree, so you can review diffs and test results independently without conflicts.
  </Step>
</Steps>

## Task Management Patterns

<Steps>
  <Step title="Draft with intent">
    Use `‚åòN` to capture requirements, architecture decisions, and acceptance criteria as the task‚Äôs Spec artifact before stage runs begin.
  </Step>

  <Step title="Refine centrally">
    Iterate on the Spec artifact in the orchestrator, and document expectations in `AGENTS.md` or the artifact itself.
  </Step>

  <Step title="Cancel and recreate when needed">
    Whenever results miss the mark, cancel the task. Its archived artifacts stay available so you can spin up a fresh task with sharper context.
  </Step>

  <Step title="Capture insights when agents stall">
    If an agent uncovers new requirements or research but can‚Äôt finish the implementation, ask it (via MCP) to create a fresh task with that new context‚Äîor have it emit a markdown summary you paste into a new task yourself. Restarting from a clean task keeps the knowledge while giving the next agent a deterministic starting point.
  </Step>
</Steps>
