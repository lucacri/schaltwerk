The orchestrator is a singleton non-task surface that runs directly in your primary repository branch (often `main`, but it can be a dedicated feature hub). It doesn’t create a worktree; instead it gives you a control room for coordinating agents and managing complex workflows.

> Any commands you run in the orchestrator’s terminals modify that branch immediately. Use tasks when you want a disposable sandbox.

## Core Capabilities

### 1. Plan and Create Tasks

Draft multiple tasks for complex features:

<Steps>
  <Step title="Open orchestrator">
    Press `⌘1` to switch to orchestrator
  </Step>

  <Step title="Create tasks">
    Use `⌘N` to create new tasks. The prompt becomes the task’s Spec artifact and includes:
    - Feature requirements
    - Implementation approach
    - Acceptance criteria
    - Dependencies
  </Step>

  <Step title="Iterate on the Spec artifact">
    Edit the task’s Spec artifact directly to refine context, constraints, and examples before stage runs begin.
  </Step>

  <Step title="Organize tasks">
    Group related tasks together

    Plan parallel work streams
  </Step>
</Steps>

### 2. Launch Agents in Parallel

Start multiple agents working on different tasks simultaneously:

<Tabs>
  <Tab title="Manual Launch">
    1. Create the first task with `⌘N`
    2. Create the next task with `⌘N`
    3. Switch between tasks with `⌘1-9`
  </Tab>

  <Tab title="MCP Automation">
    ```javascript
    // Via MCP REST API or Claude
    const tasks = [
      { name: 'auth-login', prompt: 'Implement login...' },
      { name: 'auth-register', prompt: 'Implement registration...' },
      { name: 'auth-reset', prompt: 'Implement password reset...' }
    ];

    for (const task of tasks) {
      await createTask(task.name, task.prompt);
    }
    ```
  </Tab>
</Tabs>

### 3. Review Agent Work

Check diffs, run tests, provide feedback:

<AccordionGroup>
  <Accordion title="Inspect Changes">
    - Switch to task with `⌘2-9`
    - View diffs in right panel
    - Check git status in bottom terminal
    - Review file changes
  </Accordion>

  <Accordion title="Run Tests">
    - Focus bottom terminal with `⌘/`
    - Execute test suite
    - Or use Run Mode with `⌘E`
    - Verify all tests pass
  </Accordion>

  <Accordion title="Provide Feedback">
    - Focus agent terminal with `⌘T`
    - Send instructions or corrections
    - Agent receives feedback and continues
  </Accordion>
</AccordionGroup>

### 4. Integrate Changes

Merge approved tasks back to the base branch:

<Steps>
  <Step title="Verify each task">
    - Tests pass
    - No merge conflicts
    - Changes are correct
  </Step>

  <Step title="Merge to main">
    From task header actions:
    - Create GitHub PR
    - Merge directly
    - Cherry-pick specific commits
  </Step>

  <Step title="Clean up">
    After successful merge, press `⌘D` to cancel the task
  </Step>
</Steps>

### 5. Coordinate Workflows

Chain work together for complex automation:

<CodeGroup>
```markdown Example: Feature Breakdown
Orchestrator receives: "Build complete authentication system"

1. Break into tasks:
   - Task 1: User registration with email validation
   - Task 2: Login with JWT tokens
   - Task 3: Password reset via email
   - Task 4: Session management

2. Launch agents on each task in parallel

3. Monitor progress:
   - Check which tasks are complete
   - Identify any blocked tasks
   - Provide additional context if needed

4. Review completed work:
   - Run integration tests
   - Check for conflicts between tasks
   - Verify all acceptance criteria met

5. Integrate in order:
   - Merge dependencies first (e.g., session management)
   - Then merge features (registration, login, reset)
   - Run full test suite after each merge
```

```javascript Example: Bug Triage
Orchestrator receives: "Fix all failing tests"

1. Identify failing tests:
   - Run test suite
   - Parse output for failures
   - Group by module/component

2. Create a task for each failure:
   - Task’s Spec artifact includes test output
   - Relevant file paths
   - Error messages

3. Assign agents to fix in parallel

4. Collect results:
   - Monitor which fixes are ready
   - Run tests to verify fixes
   - Merge passing fixes

5. Iterate if needed:
   - Re-run tests
   - Create new tasks for remaining failures
```
</CodeGroup>

## Best Practices

<AccordionGroup>
  <Accordion title="Plan Before Launching">
    - Draft all task Spec artifacts first
    - Identify dependencies
    - Determine execution order
    - Consider resource constraints (don't run 20 agents at once)
  </Accordion>

  <Accordion title="Name Tasks Clearly">
    Use descriptive names that indicate:
    - What the task does
    - Which component it affects
    - Relationship to other tasks

    Examples:
    - `auth-login-implementation`
    - `api-user-endpoints`
    - `fix-email-validation-bug`
  </Accordion>

  <Accordion title="Monitor Resource Usage">
    Each slot session creates 3 OS processes:
    - Shell process
    - Agent process
    - Terminal backend

    Don't run too many slot sessions simultaneously
  </Accordion>

  <Accordion title="Review Frequently">
    Don't let tasks pile up at the merge stage:
    - Review daily if possible
    - Merge or provide feedback
    - Keep the pipeline moving
  </Accordion>

  <Accordion title="Document Workflows">
    Create prompt templates for:
    - Common feature patterns
    - Bug fix procedures
    - Review checklists
    - Integration strategies
  </Accordion>
</AccordionGroup>

## Common Orchestrator Workflows

### Feature Development

<Steps>
  <Step title="Receive feature request">
    User describes desired functionality
  </Step>

  <Step title="Break into components">
    Create tasks for each component:
    - Backend API
    - Frontend UI
    - Database migrations
    - Tests
  </Step>

  <Step title="Launch agents">
    Start each task

    Agents work in parallel
  </Step>

  <Step title="Monitor progress">
    Check which tasks are complete

    Provide feedback as needed
  </Step>

  <Step title="Integrate">
    Merge components in dependency order

    Run integration tests
  </Step>

  <Step title="Final review">
    Test complete feature

    Verify all requirements met
  </Step>
</Steps>

### Bug Fixing

<Steps>
  <Step title="Identify bugs">
    Run test suite or review issue tracker
  </Step>

  <Step title="Create bug tasks">
    One task per bug with Spec artifact containing:
    - Reproduction steps
    - Expected behavior
    - Error messages
    - Related files
  </Step>

  <Step title="Assign agents">
    Start each task

    Agents investigate and fix
  </Step>

  <Step title="Verify fixes">
    Run tests for each fix

    Check for regressions
  </Step>

  <Step title="Merge fixes">
    Integrate passing fixes

    Re-run full test suite
  </Step>
</Steps>

### Refactoring

<Steps>
  <Step title="Identify refactoring needs">
    Code smell, performance issue, or architectural change
  </Step>

  <Step title="Plan refactoring">
    Break into safe, incremental steps:
    - Extract methods
    - Rename variables
    - Restructure modules
    - Update tests
  </Step>

  <Step title="Create a task for each step">
    Small, focused changes

    Each step maintains functionality
  </Step>

  <Step title="Execute incrementally">
    Complete one step at a time

    Verify tests pass after each
  </Step>

  <Step title="Final verification">
    Run full test suite

    Check performance metrics
  </Step>
</Steps>

## Limitations

<Warning>
  **Be aware of these orchestrator limitations:**
</Warning>

- **No Worktree** - Orchestrator runs on the primary branch, not in isolation
- **Resource Usage** - Each slot session consumes system resources (3 processes)
- **Concurrent Agents** - Don't run too many agents simultaneously
- **Manual Merge** - Merging tasks still requires manual review and approval
